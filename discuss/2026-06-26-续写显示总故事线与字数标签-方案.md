## 一、背景与问题

InkAI 把长篇小说拆成两个独立模式：

- **创作模式**：标签 → 人物 → 总故事线 → 知识图谱 → **第 1 章** → 创作完成。只写第一章。
- **续写模式**（`is_continuation=True`）：从第 2 章起，逐章续写。一旦开启会写进 `workflow_context.json` 永久生效。

由此带来两点诉求：

1. **续写模式看不到全书总故事线**。续写页「续写故事线生成」步骤要手动点开，且里面显示的是「下一章细纲」（`next_chapter_storyline`），不是全书总故事线（`overall_storyline`）；而该细纲常常为空，用户连入口都找不到，导致「故事线找不到」。总故事线其实完好存于 `data/novels/<id>/storyline.json`。
2. **创作阶段无法把握全书总体字数**。现有标签只有「类型/主题/风格/受众」4 类，缺一个表达篇幅体量的维度。

## 二、需求确认（已与用户拍板）

| 项 | 决策 |
|---|---|
| 需求 1 展示位置 | 续写页**常驻一块可折叠卡片**「全书总故事线」，不依赖隐藏步骤 |
| 需求 1 触发条件 | 凡续写模式的书必来自创作模式、必有 `storyline.json`，故**始终渲染**；缺失则优雅隐藏 |
| 需求 1 默认状态 | **默认折叠**（标题常驻，点击展开），不压住「写下一章」按钮 |
| 需求 2 粒度 | **篇幅档位**（5 档，见下） |
| 需求 2 选法 | **单选**（一本书一个档位） |
| 需求 2 作用 | **标记 + 影响生成**（注入 storyline 生成提示词） |

篇幅档位（5 档）：

| 档位 | 字数区间 |
|---|---|
| 短篇 | <10 万字 |
| 中篇 | 10–30 万字 |
| 长篇 | 30–80 万字 |
| 超长篇 | 80–200 万字 |
| 巨著 | >200 万字 |

## 三、需求 1 设计 · 续写页常驻「全书总故事线」卡片

**只动前端，不碰后端。** 复用已有接口 `GET /api/novels/<id>/data/storyline`（实测完整返回 `overall_storyline`）。

落点：续写页渲染函数 `displayContinuationWorkflow`（`frontend/app.js:1160`）。在「小说信息卡」之后、「续写步骤列表」之前，插入一块可折叠卡片容器；`container.innerHTML` 赋值后再异步拉取 `/data/storyline` 填充。

- 新增只读渲染函数 `renderOverallStorylineCard(storyline)`：输出主要目标、核心冲突（外/内/人际）、世界观设定（时代/地点/社会/氛围）、三幕剧（act1/act2/act3）、主题、基调、目标受众。
- 新增异步装填函数 `loadOverallStorylineIntoCard(novelId)`：fetch `/data/storyline`，成功且含 `overall_storyline` 则填入卡片正文，否则隐藏整张卡片。
- 折叠交互：Bootstrap collapse，默认 `collapsed`。
- 不改动原有「续写故事线生成」步骤（`loadContinuationStorylineDetails`），下一章细纲逻辑保持原样。

**改动文件**：`frontend/app.js`（+ 必要时 `frontend/styles.css` 加卡片样式）。

## 四、需求 2 设计 · 创作模式新增「字数标签」

顺着现有**动态标签系统**走（前端 `/config/tags` 动态遍历分类渲染，校验 `_validate_and_fix_tags` 动态遍历 `config.TAG_CATEGORIES`，`selected_tags` 默认取 `recommended_tags`），所以新分类**自动出现并流转**。**单选**通过「后端权威截断 + 前端编辑小改动」双保险实现。

后端改 3 处：

1. **`config.py`** — `TAG_CATEGORIES` 增加：
   ```python
   "字数标签": ["短篇（<10万字）", "中篇（10-30万字）", "长篇（30-80万字）", "超长篇（80-200万字）", "巨著（>200万字）"]
   ```
   注意：`_validate_and_fix_tags` 的兜底默认取该类**第一项**（短篇）。按篇幅升序排列符合下拉直觉；正常路径由 LLM 推荐对应档位，兜底仅在 LLM 返回非法值时触发（边缘情况，可接受）。

2. **`agents/tag_selector.py`** — 两处硬编码 4 键需补成 5 键：
   - 推荐提示词的返回 JSON 模板（约 53–57 行）加 `"字数标签": ["推荐的字数标签"]`，并在推荐原则里加一条「依据用户对篇幅/体量的描述推荐档位，无明确描述时默认『长篇（30-80万字）』」。
   - `_get_default_tags`（约 85–90 行，空需求兜底）加 `"字数标签": ["长篇（30-80万字）"]`。
   - `_format_tag_categories` 动态遍历，无需改。
   - **单选权威兜底**：`_validate_and_fix_tags` 在动态遍历后，对「字数标签」截断为只保留第 1 个有效项（`valid_tags["字数标签"] = valid_tags["字数标签"][:1]`），保证落盘数据恒为单值。

3. **`agents/storyline_generator.py`** — `_generate_overall_storyline` 的提示词「创作背景」段（约 42–48 行）注入目标篇幅，并在「故事设计要求」加一条：*依据目标篇幅【X】规划三幕剧的剧情体量、支线数量与节奏密度（短篇紧凑单线、长篇及以上多线铺陈）*。新增小工具从 tags 中取出字数标签拼进提示词。

**前端单选小改动**：标签编辑弹窗（`frontend/app.js` 约 4125–4210 的「添加标签」逻辑）对「字数标签」分类做「新增即替换」——选该类时先清空已有再加入，避免出现「中篇+巨著」并存。后端截断为权威保证，前端改动只为交互体验。

**改动文件**：`config.py`、`agents/tag_selector.py`、`agents/storyline_generator.py`、`frontend/app.js`（编辑弹窗单选）。

## 五、影响面与风险

- 需求 1 纯前端增量，复用现成接口，不改后端/数据结构，对续写流程零破坏；唯一风险是卡片渲染异常，已用「缺失即隐藏」兜底。
- 需求 2 顺动态标签链路，新分类对**存量小说**：其 `tags.json` 没有「字数标签」键，前端 `Object.entries(selected_tags)` 只是少渲染一类，不报错；编辑弹窗里可手动补选。新建小说自动带上。
- 不引入新依赖，不改动数据落盘格式（仅 tags 多一个键）。
